# 要件定義書

## 1. システム概要
本システムは、自身と友人5名程度が利用する小規模Minecraftサーバーを、ローカル環境からクラウド環境へ段階的に移行しながら運用するためのインフラ基盤である。サーバーの構築、起動停止、バックアップ、監視、復旧、設定管理を通じて、実運用に近い保守業務を経験できることを目的とする。

## 2. 機能要件
### 2.1 利用者向け機能
* 利用者は、認証されたメンバーのみMinecraftサーバーへ接続できること。
* 利用者は、サーバー稼働中に通常のMinecraftプレイを行えること。

### 2.2 管理者向け機能
* 管理者は、サーバーの起動および停止を実行できること。
* 管理者は、サーバー状態を確認できること。
* 管理者は、バックアップの取得状況および復旧手順を確認できること。
* 管理者は、Minecraftのホワイトリストを管理できること。

### 2.3 自動運用機能
* サーバー停止時に自動バックアップを取得できること。
* 日次で自動バックアップを取得できること。
* バックアップファイルの保持世代を管理し、一定数を超えた古いバックアップを自動削除できること。
* 障害発生時に管理者へ通知できること。
* 一定時間プレイヤーが不在の場合にサーバーを自動停止できること。

### 2.4 将来拡張機能
* 将来的にDiscord連携による起動停止、状態確認、通知、ホワイトリスト連携を追加できる構成とすること。
* 将来的にForgeを利用した拡張に対応しやすい構成とすること。

## 3. 非機能要件
### 3.1 可用性・復旧性
* 障害発生時に、バックアップから速やかに復旧できる手順を用意すること。
* 定期バックアップと停止時バックアップにより、復旧に必要なデータを確保できること。

### 3.2 性能
* 利用者5名程度が同時接続しても、通常プレイに支障が出ないこと。
* サーバー起動後、短時間で接続可能な状態になること。

### 3.3 保守性
* 構築手順、運用手順、障害対応手順を文書化し、再現可能な形で管理すること。
* 自動化処理はシェルスクリプト等で整理し、修正しやすい構成にすること。

### 3.4 セキュリティ
* サーバーは認証された利用者のみ接続できること。
* 管理操作は管理者のみに限定すること。
* 公開範囲は友人グループ内に限定し、不特定多数へ一般公開しないこと。

### 3.5 拡張性
* ローカル環境からクラウド環境への移行を前提として、構成変更に対応しやすいこと。
* 将来のDiscord連携やForge導入を妨げない設計余地を持つこと。

### 3.6 運用性
* バックアップ世代の保持数を管理できること。
* 監視通知は、発生時に確認可能な時間内に管理者へ届くこと。

## 4. 制約条件
* 利用者は自身と友人5名程度に限定する。
* 一般公開は行わない。
* まずはローカル環境で構築し、その後クラウドへ移行する。
* Minecraft本体やワールドデータはGit管理の対象外とする。
* Discord連携などの具体的な実装方法は、要件定義書ではなく基本設計書以降で決定する。

## 5. フェーズ分割
* フェーズ1では、ローカル環境でのMinecraftサーバー構築、運用自動化、バックアップ、監視、復旧、クラウド移行を完了させる。
* フェーズ2以降で、Discord連携や自動停止条件の拡張などの発展機能を追加する。
